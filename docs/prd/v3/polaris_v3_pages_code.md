# 《北极星》V3 研发任务拆分稿 v0.1

---

# 1. V3 研发目标与交付边界

## 1.1 V3 必须交付的能力

V3 完成后，用户应可以完成以下升级闭环：

* 查看请求 / Mock / 集合等资产的历史版本
* 对比两个版本的差异
* 对比两次集合运行的差异
* 从请求或运行结果创建问题现场
* 查看和管理问题现场
* 生成轻量只读分享
* 在项目页看到更完整的摘要信息
* 通过 MCP 向外部 AI / Agent 暴露问题现场、运行历史、版本信息和项目摘要

---

## 1.2 V3 不纳入本轮开发的能力

以下内容明确不纳入 V3：

* 内置 AI 助手
* 模型调用
* 产品内对话入口
* 第三方插件市场
* 第三方扩展安装
* 企业级复杂权限治理
* 团队实时协同编辑
* 重型自动化测试平台
* 高门槛脚本系统

---

# 2. V3 技术模块拆分

V3 继续沿用 V1 / V2 的四个模块，并新增以下重点服务：

## 模块 A：Local Core（增强）

新增：

* Version Service
* Diff Service
* Issue Bundle Service
* Share Service
* Summary Service

## 模块 B：Web Console（增强）

新增：

* 问题现场页
* 问题现场详情页
* 版本查看 / 对比页
* 运行对比视图
* 分享弹层
* 项目摘要增强模块

## 模块 C：Browser Extension（极轻增强）

可选新增：

* 最近项目快捷入口
* 最近问题现场快捷入口

## 模块 D：Shared Contract / Schema（增强）

新增：

* AssetVersion
* IssueBundle
* ShareRecord
* ProjectSummary
* Diff 结果结构
* V3 级 MCP tools / resources 协议

# 3. 模块 A：Local Core 研发任务拆分

---

## A-1 Version Service（历史版本服务）

### 目标

为请求、Mock、集合等核心资产提供历史版本记录与查询能力。

### 任务

* A-1-1 AssetVersion 数据结构定义
* A-1-2 版本快照写入机制
* A-1-3 请求版本记录实现
* A-1-4 Mock 版本记录实现
* A-1-5 集合版本记录实现（可选，建议纳入）
* A-1-6 版本列表查询接口
* A-1-7 单个版本详情接口
* A-1-8 最近变更资产摘要接口

### 产出

* 请求 / Mock / 集合可查询历史版本
* 项目详情可读取最近变更摘要

### V3 范围建议

V3 先做“记录 + 查看 + 对比”，不做“回滚恢复”。

---

## A-2 Diff Service（差异对比服务）

### 目标

提供请求版本对比和运行结果对比能力。

### 任务

* A-2-1 Request diff 结构定义
* A-2-2 Mock diff 结构定义
* A-2-3 Run diff 结构定义
* A-2-4 两个请求版本对比接口
* A-2-5 两次运行结果对比接口
* A-2-6 差异摘要生成逻辑
* A-2-7 字段级差异高亮数据结构
* A-2-8 Body 差异摘要生成

### 产出

* 可对比两个版本的请求 / Mock
* 可对比两次集合运行

### V3 范围约束

先做：

* 字段差异
* 响应摘要差异
* 成功 / 失败项变化

不做：

* 复杂可视化 diff 编辑器
* 三方对比
* 自动合并

---

## A-3 Issue Bundle Service（问题现场服务）

### 目标

将请求、运行、环境、Mock、规则等上下文打包成问题现场对象。

### 任务

* A-3-1 IssueBundle 数据结构定义
* A-3-2 从请求创建问题现场接口
* A-3-3 从运行记录创建问题现场接口
* A-3-4 手动创建空白问题现场接口
* A-3-5 问题现场列表接口
* A-3-6 问题现场详情接口
* A-3-7 问题现场更新接口
* A-3-8 问题现场删除接口
* A-3-9 复制问题现场接口
* A-3-10 环境快照写入
* A-3-11 代理规则快照写入
* A-3-12 关联请求 / 运行 / Mock 关系写入

### 产出

* 用户可创建、查看、复制、删除问题现场
* 问题现场能承载最关键排障上下文

---

## A-4 Run History Service 升级（支持对比）

# 11. 研发风险点

---

## 风险 1：V3 概念变多导致产品变重

需要继续坚持通俗表达：

* 问题现场
* 这次和上次哪里不同
* 最近变更
* 分享给同事查看

不要引入太多平台化术语。

---

## 风险 2：版本记录范围膨胀

建议 V3 只覆盖：

* 请求
* Mock
* 集合
  不要把所有对象都强制版本化。

---

## 风险 3：运行对比容易走向测试平台

V3 只做“帮助理解差异”，不做复杂测试分析系统。

---

## 风险 4：问题现场快照结构复杂

环境快照和规则快照要足够实用，但不能为了完整性把实现做得过重。

---

## 风险 5：分享涉及安全性

V3 建议只做轻量只读分享，并预留后续脱敏 / 权限增强空间。

---

## 风险 6：MCP 能力继续膨胀

V3 的 MCP 仍然只围绕“查询上下文”和“触发基础对象创建”，不要走向复杂代理编排。

---
