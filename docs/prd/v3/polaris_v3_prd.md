# 《北极星》V3 基础 PRD v0.1

---

## 1. 文档信息

### 1.1 文档名称

《北极星》V3 产品需求文档 PRD v0.1

### 1.2 产品名称

**中文名：北极星**
**英文名：Polaris**

### 1.3 一句话定义

**Polaris：一个以本地代理为核心的接口资产工作台，支持请求查看、保存、重放、Mock、项目管理、运行记录与问题复现，并通过 MCP 向外部 AI / Agent 暴露完整的本地接口能力。**

### 1.4 V3 版本目标

在 V1、V2 的基础上，V3 的目标是进一步把北极星升级为一个**可长期沉淀、可复现问题、可轻协作、可被外部 AI 深度使用的本地接口资产平台**。

V3 不改变底层定位，不新增模型层，而是在以下几个方向增强：

* 资产沉淀
* 运行记录与对比
* 问题现场复现
* 轻协作与分享
* 更完整的 MCP 暴露层
* 扩展机制继续完善

---

# 2. V3 版本背景

V1 解决的是“能开始用”的问题，V2 解决的是“能围绕项目持续用”的问题。
到了 V3，用户会开始出现下面这些需求：

* 请求、Mock、集合和运行记录需要长期沉淀
* 同一个问题希望被打包复现，而不是靠口头描述
* 用户希望快速对比两次运行差异
* 项目内资产需要有基础版本感知
* 需要更轻量的分享与协作
* 外部 AI / Agent 不只需要单次请求，还需要项目上下文、运行历史和问题上下文

因此 V3 的重点不是“增加更多零碎功能”，而是建立一个更完整的**本地接口资产闭环**。

# 3. V3 产品目标与非目标

## 3.1 产品目标

### 目标 1：建立长期资产沉淀能力

请求、Mock、集合、运行记录都能长期保留和复用。

### 目标 2：建立问题复现能力

用户可以把一次失败请求、运行结果、相关 Mock 和环境信息打包成“问题现场”。

### 目标 3：建立运行对比能力

用户可以快速看到两次运行、两条请求或两个响应的差异。

### 目标 4：建立轻协作能力

支持以只读和分享为主的轻量协作，不引入重型组织体系。

### 目标 5：建立更完整的 MCP 上下文暴露

让外部 AI / Agent 可以读取问题现场、历史版本、运行记录和项目摘要。

### 目标 6：继续控制复杂度

V3 功能虽然更多，但页面仍然要尽量通俗、低门槛，不走重型平台路线。

---

## 3.2 非目标

V3 明确不做：

* 内置 AI 助手
* 模型调用能力
* 产品内 AI 对话入口
* 重型自动化测试平台
* 高门槛脚本系统
* 复杂插件市场
* 企业级复杂权限治理
* 团队实时多人协同编辑
* 动态脚本式全流程编排引擎

---

# 4. V3 核心定位

## 4.1 产品定位

**一个本地运行的接口资产平台。**

它解决的问题是：

* 如何长期保存和复用项目里的接口资产
* 如何快速复现一个接口问题
* 如何对比两次运行差异
* 如何向外部 AI / Agent 提供更完整的接口上下文

# 5. 目标用户

## 5.1 核心用户

* 前端研发
* 后端研发
* 测试研发
* 联调负责人
* 技术支持 / 排障人员
* 需要把本地接口能力接入 AI / Agent 的工程师

## 5.2 用户特征

* 已经习惯使用北极星 V1 / V2
* 开始有长期维护项目资产的需求
* 不满足于“单次调试”
* 希望用更低成本复现和分享问题
* 需要对运行结果做简单对比
* 希望外部 AI 能基于更完整上下文工作

---

# 6. V3 核心使用场景

## 场景 1：围绕项目持续沉淀接口资产

用户长期在某个项目中保存请求、维护 Mock、管理集合和运行记录。

## 场景 2：快速打包一个问题现场

用户把失败请求、当时使用的环境、命中的 Mock 和运行结果打包，发给其他同学或供外部 AI 读取。

## 场景 3：对比两次运行差异

用户希望知道同一个集合这次为什么失败、和上次相比差在哪里。

## 场景 4：对比两条请求或两个响应

用户想快速确认参数变化、响应结构变化和状态变化。

## 场景 5：分享接口资产给其他人查看

用户可以分享某个请求、某个 Mock、某次运行记录或某个问题现场。

## 场景 6：让外部 AI / Agent 获取完整上下文

外部 AI / Agent 不只是读取单条请求，还可以读取项目摘要、运行记录、问题现场和版本信息。

# 15. 非功能需求

## 15.1 易用性

* 页面变多但认知负担不能明显上升
* 问题现场和运行对比要尽量通俗
* 不使用复杂测试平台术语

## 15.2 性能

* 运行记录列表查询流畅
* 版本查看和对比响应及时
* 问题现场详情打开可接受

## 15.3 稳定性

* 新增历史和分享能力不能影响代理核心稳定性
* 问题现场创建失败不影响原始资产
* MCP 高级能力读取稳定

## 15.4 安全

* 仍以本地存储为主
* 分享时应支持只读
* 后续需考虑脱敏策略预留
* 不开放第三方扩展安装

# 19. 风险与注意事项

## 风险 1：功能变多导致产品“变重”

必须继续坚持通俗、低门槛的设计原则。

## 风险 2：问题现场概念太抽象

需要通过清晰文案和示例让用户理解“问题现场”的价值。

## 风险 3：运行对比演变成复杂测试系统

要坚持“帮助理解差异”，不要走向重型测试平台。

## 风险 4：分享涉及安全风险

需要在后续版本中考虑脱敏和分享控制策略。

## 风险 5：MCP 能力继续膨胀

只围绕项目资产、运行历史和问题上下文扩展，不做复杂 agent orchestration。
