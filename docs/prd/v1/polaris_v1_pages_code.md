
# 《北极星》V1 研发任务拆分稿 v0.1

---

# 1. V1 研发目标与交付边界

## 1.1 V1 必须交付的能力

V1 研发完成后，用户应可以完成以下完整闭环：

* 本地启动北极星服务
* 浏览器插件可切换代理模式
* 用户可以看到真实请求
* 用户可以查看请求详情
* 用户可以保存请求
* 用户可以重放请求
* 用户可以创建基础 Mock
* 一个接口支持多个 Mock 方案
* 可以通过 MCP 读取请求和调用基础工具
* 系统内部预留扩展机制，但不开放第三方安装

---

## 1.2 V1 不纳入本轮开发的能力

为了避免范围膨胀，以下内容明确排除：

* 项目体系
* 环境变量
* 请求集合
* 运行记录
* 历史版本
* 问题现场
* 分享能力
* 内置 AI
* 第三方插件市场
* 团队协作能力

---

# 2. 技术模块拆分

V1 建议拆成 4 个研发模块：

## 模块 A：Local Core

本地服务核心，负责代理、存储、Mock、MCP、本地 API。

## 模块 B：Web Console

主工作台界面，负责请求展示、请求管理、Mock 管理、调试、设置。

## 模块 C：Browser Extension

浏览器插件，负责代理模式切换、当前站点规则快捷写入、打开控制台。

## 模块 D：Shared Contract / Schema

负责前后端共享的数据结构、接口协议、状态枚举、MCP tool / resource 定义。

这个 D 模块虽然不是独立产品模块，但在实际研发里非常关键，不先定协议，后面会反复改。

---

# 3. 模块 A：Local Core 研发任务拆分

---

## A-1 本地服务基础框架

### 目标

搭建本地常驻服务，具备启动、关闭、配置加载、状态检查能力。

### 任务

* A-1-1 本地服务启动入口
* A-1-2 配置读取与默认配置初始化
* A-1-3 健康检查接口
* A-1-4 服务状态对象定义
* A-1-5 日志基础能力

### 产出

* 可启动本地服务进程
* 可返回服务在线状态
* 可读取基础配置（端口、MCP 开关等）

---

## A-2 Proxy Engine（代理核心）

### 目标

支持本地代理端口监听，并捕获 HTTP / HTTPS 请求。

### 任务

* A-2-1 本地代理端口监听
* A-2-2 HTTP 请求转发
* A-2-3 HTTPS 抓包支持基础接入
* A-2-4 请求与响应数据采集
* A-2-5 错误请求处理
* A-2-6 代理模式与规则输入接口预留

### 产出

* 代理服务可用
* 请求和响应可被采集
* 能为 Web Console 提供实时请求数据

### 注意

HTTPS 证书能力至少要具备最小可用链路，但 V1 说明和安装引导可以先简单。

---

## A-3 Traffic Recorder（流量记录）

### 目标

把捕获到的请求结构化存储，支持实时查看和详情读取。

### 任务

* A-3-1 RequestRecord 数据结构定义
* A-3-2 请求列表缓存 / 存储
* A-3-3 请求详情读取接口
* A-3-4 搜索 / 筛选能力基础支持
* A-3-5 列表分页或数量限制策略
* A-3-6 大响应体截断 / 存储策略

### 产出

* 可获取实时请求列表
* 可查看某条请求完整详情
* 支持关键词 / method / statusCode 基础筛选

---

## A-4 Saved Request（请求保存与重放）

### 目标

支持将真实请求保存为资产，并可重放。

### 任务

* A-4-1 SavedRequest 数据结构定义
* A-4-2 保存请求接口
* A-4-3 查询已保存请求列表接口
* A-4-4 更新请求接口
* A-4-5 删除请求接口
* A-4-6 请求重放接口
* A-4-7 手动调试请求执行接口

### 产出

* 请求可从实时流量保存
* 已保存请求可管理
* 已保存请求可重放
* 调试页可手动发起请求

---

## A-5 Mock Engine（基础 Mock）

### 目标

支持基础固定返回 Mock，并支持一个接口多个 Mock 方案。

### 任务

* A-5-1 MockRule 数据结构定义
* A-5-2 MockRule CRUD
* A-5-3 Mock 启用 / 禁用
* A-5-4 请求匹配逻辑（Method + URL）
* A-5-5 同接口多个 Mock 方案的数据模型支持
* A-5-6 V1 生效策略定义
* A-5-7 命中次数统计
* A-5-8 最近命中时间记录

### 产出

* 可创建多个 Mock 方案
* Mock 可启停
* Mock 命中可替代真实返回
* 命中统计可被界面读取

### V1 需要先定的规则

建议 V1 先采用：

* 同一 Method + URL 可存在多个方案
* 同时只允许一个方案为“启用态”
* 若启用新方案，则提示关闭其他方案或自动互斥

这样能明显降低 V1 匹配复杂度。

---

## A-6 Proxy Rule / PAC 支持

### 目标

为浏览器插件提供代理模式和规则支撑。

### 任务

* A-6-1 代理模式枚举定义
* A-6-2 ProxyRule 数据结构定义
* A-6-3 当前站点添加 / 删除规则接口
* A-6-4 当前代理模式查询接口
* A-6-5 代理规则列表读取接口
* A-6-6 PAC 内容生成接口或规则输出接口
* A-6-7 插件调用的轻量控制 API

### 产出

* 插件可切换代理模式
* 插件可对当前站点开启代理
* Core 能返回规则配置结果

---

## A-7 MCP Server

### 目标

提供 V1 的基础 MCP tool / resource。

### 任务

* A-7-1 MCP Server 基础框架
* A-7-2 tool 注册机制
* A-7-3 resource 注册机制
* A-7-4 V1 tool 实现：

  * list_requests
  * get_request_detail
  * save_request
  * replay_request
  * create_mock_rule
  * enable_mock_rule
  * run_request
* A-7-5 V1 resources 实现：

  * 最近请求列表
  * 已保存请求列表
  * Mock 列表
  * 当前代理模式
  * 当前代理规则列表
* A-7-6 MCP 开关与配置读取

### 产出

* MCP 服务可被外部客户端连接
* 基础 tools / resources 可用

---

## A-8 Storage Adapter（本地存储与预留扩展）

### 目标

以可替换存储适配层承接 V1 数据读写。

### 任务

### 任务

# 8. 前后端 / 插件 / Core 分工建议

建议负责：

* 代理
* 流量
* 存储

建议负责：

* 所有页面
* 组件系统
* 状态管理
* 与本地 API 联调
* 交互和空态

适合偏前端同学承接。

## 8.3 Browser Extension 负责人

建议负责：

* 插件弹窗
* 浏览器代理模式控制

建议由 1 名主负责人牵头，和 Core、Console、Extension 一起定。

# 11. 研发风险点

---

## 风险 1：代理与 HTTPS 能力开发复杂度高

这是 V1 最大技术风险。
建议尽早完成最小可用链路验证，不要等 UI 全做完才发现代理底座不稳。

---

## 风险 2：插件与本地服务联动不稳定

特别是：

* 状态同步
* 当前站点规则写入
* 代理模式切换反馈

建议尽早拉通插件 -> Core 的最短链路。

---

## 风险 3：Mock 多方案逻辑容易变复杂

建议 V1 先采用“同一 Method + URL 下仅允许一个启用方案”的约束，不做复杂匹配优先级。

---

## 风险 4：页面容易越做越专业

前端实现时要注意：

* 默认信息层级
* 空态文案
* 主按钮数量
* 字段密度

---

## 风险 5：MCP 范围膨胀

V1 只做基础 tools / resources，先保证稳定，不要扩太多。

---
