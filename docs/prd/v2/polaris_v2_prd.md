# 《北极星》V2 基础 PRD v0.1

---

## 1. 文档信息

### 1.1 文档名称

《北极星》V2 产品需求文档 PRD v0.1

### 1.2 产品名称

**中文名：北极星**
**英文名：Polaris**

### 1.3 一句话定义

**Polaris：一个以本地代理为核心的接口资产工作台，支持请求查看、保存、重放、Mock、项目管理与集合运行，并通过 MCP 向外部 AI 暴露这些能力。**

### 1.4 V2 版本目标

在 V1 已经建立的基础上，V2 的目标是把“单次请求工具”升级为“项目级接口资产工作台”，让用户可以围绕一个项目持续管理：

* 请求资产
* Mock 资产
* 代理规则
* 环境变量
* 请求集合
* 运行记录
* MCP 项目级能力

---

# 2. V2 版本背景

V1 解决的是“能不能顺畅开始用”的问题，用户已经可以完成：

* 开启代理
* 查看真实请求
* 保存请求
* 重放请求
* 创建基础 Mock
* 通过 MCP 暴露基础工具

但在持续使用过程中，会出现新的需求：

* 请求不再是零散的，而是需要按项目组织
* 请求在不同环境下需要复用
* Mock 需要按项目和场景管理
* 用户希望保存一组常用请求并批量运行
* 外部 AI / Agent 需要读取项目上下文，而不是只看单条请求
* 用户需要更稳定的规则代理管理方式

因此 V2 的核心不是新增很多零散功能，而是建立“项目维度的资产组织能力”。

---

# 3. V2 产品目标与非目标

## 3.1 产品目标

### 目标 1：建立项目级组织能力

让请求、Mock、代理规则和环境变量都能归属于项目。

### 目标 2：建立请求集合能力

让用户可以把多个请求组织成一个集合，便于批量运行和重复使用。

### 目标 3：建立环境变量能力

支持在开发、测试、预发等不同环境间切换请求配置。

### 目标 4：增强 Mock 的可管理性

支持一个接口有多个 Mock 方案，并按项目归类管理。

### 目标 5：增强 MCP 的项目级暴露能力

让外部 AI / Agent 能以“项目”为上下文读取资产，而不是只操作单条请求。

### 目标 6：继续保持低门槛体验

即使功能增强，页面仍然要保持通俗，不做重型平台感。

---

## 3.2 非目标

V2 明确不做：

* 内置 AI 助手
* 模型推理能力
* 对话式 AI 工作区
* 重型自动化测试平台
* 高级断言 DSL
* 动态脚本式 Mock 平台
* 团队实时协作编辑
* 企业审批与权限流
* 第三方插件市场
* 第三方扩展安装机制

---

# 4. V2 核心定位

V2 的 Polaris 不再只是“本地抓包工具 + Mock 工具”，而是：

## 4.1 产品定位

**一个本地运行的项目级接口资产工作台。**

它解决的问题是：

* 如何围绕某个项目持续沉淀请求资产
* 如何在不同环境下复用请求
* 如何管理项目里的 Mock 和代理规则
* 如何让外部 AI / Agent 获得项目上下文

---

# 5. 目标用户

## 5.1 核心用户

* 前端研发
* 后端研发
* 测试研发
* 联调负责人
* 需要将本地接口能力接入 AI / Agent 的工程师

## 5.2 用户特征

* 已具备 V1 使用能力
* 不满足于单条请求的临时操作
* 开始有项目组织需求
* 开始需要多环境切换
* 希望请求、Mock、规则有归属感和可管理性

---

# 6. V2 核心使用场景

## 场景 1：按项目管理接口资产

用户创建一个项目，把保存的请求、Mock、代理规则都收纳进去。

## 场景 2：在不同环境中复用请求

用户希望同一条请求可以在开发环境、测试环境之间切换。

## 场景 3：保存一组常用请求

用户把“登录、获取列表、获取详情”放进一个集合，后续可一键批量运行。

## 场景 4：一个接口维护多个 Mock 方案

用户为同一个接口维护正常态、空数据态、失败态等多个 Mock。

## 场景 5：让外部 AI / Agent 围绕项目工作

外部 AI / Agent 可以查询某个项目的请求、Mock、集合、环境和规则。

# 15. 非功能需求

## 15.1 易用性

* 项目能力增强后，页面不能变得像重型平台
* 用户不需要理解太多专业术语
* 主要页面继续保持“动作优先”

## 15.2 性能

* 项目切换流畅
* 集合运行结果展示及时
* 请求和 Mock 资产查询可接受

## 15.3 稳定性

* 项目能力不能影响代理核心稳定性
* 集合运行失败不影响请求资产本身
* MCP 项目级资源读取稳定

## 15.4 安全

* 仍以本地存储为主
* 不默认上传资产
* 不开放第三方扩展安装

# 19. 风险与注意事项

## 风险 1：项目化后页面复杂度上升

需要避免把“项目系统”做成重后台。

## 风险 2：环境变量引入学习成本

变量能力必须轻量，不做脚本式平台。

## 风险 3：集合运行容易演变成测试平台

要坚持“复用和验证”定位，不转向复杂测试产品。

## 风险 4：MCP 能力膨胀

只围绕项目资产做暴露，不引入复杂工作流编排。
